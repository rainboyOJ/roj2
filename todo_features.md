# 后续功能优化记录

## 高优先级

- [x] **1. 去除残留的 i18n 翻译层**

项目现在已经确定只保留中文，不再需要英文界面。但当前页面模板里仍然大量使用 `t('...')`，HTTP 渲染入口也还会注入 `createViewContext()`。

这个结构在多语言阶段是合理的，但现在会增加源码阅读和页面修改成本：

- 阅读 Pug 模板时不能直接看到最终中文文案。
- 修改一个页面文案时，需要在模板和 `view-i18n.ts` 之间来回跳转。
- 后续新增页面时，容易继续复制不必要的翻译 key。

后续建议：

- 删除 `apps/api-server/src/view-i18n.ts`。
- 在 Pug 模板中直接写中文文案。
- 删除 `createViewContext()` 注入逻辑。
- 保留 `assetUrl`、`currentUser`、`isAdminArea` 等真正需要的模板上下文。
- 更新相关测试，确保页面仍然渲染中文内容。

- [x] **2. 统一 HTML 表单错误处理**

当前 HTML 表单路由遇到错误时，很多地方直接返回纯文本，例如 `Invalid registration payload`、`Invalid problem payload`。

这个方式实现简单，但用户体验不够好：

- 用户会离开原页面，看到一行纯文本错误。
- 表单里已经填写的内容无法保留。
- 不同页面的错误展示方式不一致。

后续建议：

- 为登录、注册、个人资料、后台题目、后台语言设置、年级管理等表单统一错误显示方式。
- HTML 表单校验失败时优先回到原页面，并显示中文错误提示。
- API 路由继续返回 JSON，不和 HTML 页面混在一起。
- 前端校验只作为体验优化，后端仍然保留完整校验。

- [ ] **3. 明确删除用户的数据一致性策略**

当前删除用户时会删除 `users` 和 `sessions`，但该用户历史提交、题目进度等数据仍可能保留。

这会带来几个问题：

- 提交列表中可能还能看到已经不存在的用户数据。
- 用户题目进度表中可能留下孤立记录。
- 后续统计、排行榜、管理后台查询时，删除用户的语义不够清楚。

后续建议先确定删除策略：

- 如果用户已有提交，可以禁止硬删除，只允许禁用或软删除。
- 如果允许硬删除，需要明确是否级联删除 `sessions`、`user_problem_progress`。
- 历史 `submissions` 建议保留，因为它们属于评测记录；页面展示可继续使用提交里保存的 `username`、`displayName`。
- 后台删除操作需要给管理员明确提示，避免误删。

- [x] **11. 班级改为数据库字典并禁止用户手动填写**

注册页面当前让用户手动输入班级，容易出现“1班 / 1 班 / 一班”等不一致数据，后续按班级过滤排行榜和提交列表时会变得很难维护。

后续建议：

- 新增 `classes` 或等价集合，保存班级字典。
- 初始化默认班级为 `1 班` 到 `40 班`。
- 注册页面把班级输入框改成下拉选择，不允许自由输入。
- 管理后台增加班级管理页面，支持添加、删除或停用班级。
- 注册接口继续做后端校验，只允许选择数据库中可用班级。
- 用户资料里的班级修改也应改为下拉选择，避免绕过注册页产生脏数据。

- [x] **12. 普通用户可查看全站提交列表并支持过滤**

当前提交列表里普通用户只能看到自己的提交。后续希望普通用户也可以看到全站提交记录，但仍然要保护代码隐私：普通用户只能查看自己的提交代码，不能看到其他用户的代码。

后续建议：

- `/submissions` 对普通用户展示全站提交列表。
- 提交详情页保留权限控制：管理员可看所有代码，普通用户只能看自己的代码；查看他人提交详情时不显示源码或返回无权限。
- 提交列表增加过滤条件：
  - 按题目过滤。
  - 按用户过滤。
- API 层同步支持过滤参数，避免只在 Pug 页面做前端过滤。
- 数据库查询增加必要索引或复用已有索引，避免全站提交多时列表变慢。

- [x] **14. 题目列表和提交列表增加分页及后台页大小设置**

题目列表和提交列表后续数据量会持续增长，如果一直一次性展示全部记录，页面会变慢，也不利于用户查找。

后续建议：

- `/problems` 增加分页能力。
- `/submissions` 和后台提交列表统一支持分页。
- 分页大小由管理员后台设置，例如每页 20、50、100 条。
- 将分页大小保存到数据库站点设置中，而不是写死在路由常量里。
- 页面上保留当前筛选条件和页码，翻页时 query 参数不丢失。
- API 层同步支持 `page`、`pageSize`，并限制最大 `pageSize`，避免一次性拉取过多数据。

- [x] **15. 管理后台用户审核增加分页并修复批量审核**

管理员用户审核页面目前缺少分页，用户多时会一次性加载全部用户；同时批量审核流程疑似存在 bug，需要单独排查。

后续建议：

- `/admin/users` 增加分页能力。
- 用户审核 API 同步支持分页参数。
- 批量审核只作用于当前勾选的用户，不应误处理未选用户。
- 批量通过、批量拒绝后保留当前筛选和页码，方便继续审核。
- 增加测试覆盖：
  - 未选择用户时返回中文错误。
  - 只审核选中的用户。
  - 分页后第二页用户可以正常批量审核。
  - 批量操作失败时页面保留用户列表和错误提示。

- [ ] **18. 题目列表增加搜索和完成状态筛选**

题目数量增长后，学生只靠分页浏览题目会比较低效。当前题目列表已经有分页和完成状态展示，但还不能按题号、标题或完成状态快速定位。

后续建议：

- `/problems` 增加题号和标题搜索。
- 登录用户可以按完成状态筛选，例如全部、已通过、已尝试、未开始。
- URL 使用 query 参数保存筛选状态，刷新和分享链接时不丢失。
- 分页链接保留当前筛选条件。
- API 层同步支持相同筛选条件，避免只在前端过滤。

- [x] **19. 管理员题目管理增加搜索和可见性筛选**

后台题目管理当前只能浏览完整列表。题目变多后，管理员查找某道题、发布隐藏题目或确认题目可见性会比较慢。

后续建议：

- `/admin/problems` 增加题号和标题搜索。
- 增加可见性筛选：全部、已发布、隐藏。
- 发布题目、编辑题目后尽量保留当前筛选和分页上下文。
- 后端查询层支持筛选，避免后台页面一次性加载全部题目再前端过滤。
- 增加页面和 API 测试，确保筛选条件正确传递。

- [x] **20. 提交详情页增加返回题目和再提交入口**

提交详情页现在主要用于查看评测结果，侧边栏只有返回提交列表和打开题目列表。学生评测失败后，通常需要马上回到原题继续修改并再次提交。

后续建议：

- 提交详情页增加“返回题目”按钮，直接跳到该提交对应题目。
- 增加“再提交此题”入口，优先跳到题目提交区域。
- 评测未结束时保留自动刷新逻辑，不影响新入口。
- 普通用户无权查看他人源码时，仍然可以查看题目信息和返回题目。

## 中优先级

- [x] **4. 抽取管理后台表单解析逻辑**

当前 `apps/api-server/src/routes/admin.ts` 中，HTML 表单和 API 路由存在一些重复解析逻辑，例如题目表单、语言设置、年级表单。

这些重复代码短期可接受，但后台功能继续增加后会变得难维护：

- checkbox 字段需要反复处理 string / string[]。
- HTML 表单和 API 的 zod 校验入口不完全一致。
- 修改一个字段时，容易漏掉另一个路由分支。

后续建议：

- 抽取 `parseProblemFormBody()`。
- 抽取 `parseLanguageSettingsFormBody()`。
- 抽取 `parseGradeFormBody()`。
- HTML 路由和 API 路由尽量复用同一套 schema。
- 保持现有 URL 和行为不变，只做结构整理。

- [x] **5. 静态资源路由整理为统一 manifest**

当前静态资源路由使用手写白名单 Map，`problem-editor.js` 又单独注册了一条路由。这个方式安全，但随着本地资源增多会越来越零散。

后续建议：

- 建立统一的 asset manifest，集中声明允许访问的资源和 content-type。
- 普通 JS/CSS、editor bundle、字体资源都走一致的路径校验和缓存策略。
- 保持本地资源白名单，不开放任意文件读取。
- 保留开发环境 `no-store`、生产环境版本号缓存的行为。

- [ ] **6. 为 `quick_start_for_test.sh` 增加自动化测试**

`quick_start_for_test.sh` 是本地调试 Docker 部署服务的重要入口，但当前测试主要覆盖 `install.sh` 和 dev 脚本，快速调试脚本还缺少自动化验证。

后续建议使用 fake `docker`、fake `curl`、fake `node` 或可控脚本环境覆盖这些场景：

- 缺少 `roj-mongodb` 容器时输出警告并退出。
- 缺少 `roj-judge-server` 容器时输出警告并退出。
- 存在 `roj-api-server` 和 `roj-judge-dispatcher` 时会先停止它们。
- 能从 `docker port` 识别 judge-server 映射端口。
- 脚本退出时默认恢复被停止的 Docker 容器。
- 设置 `RESTORE_DOCKER_SERVICES=0` 时不恢复容器。

- [x] **7. 默认题目 seed 改为扫描目录**

当前默认题目初始化固定读取 `packages/db/default_problems/1000`。但目录结构已经是 `default_problems/<pid>/`，天然支持多个默认题目。

后续建议：

- 启动 seed 时扫描 `packages/db/default_problems/` 下的所有题号目录。
- 每个目录读取 `metadata.json`、`content.md` 和可选的 `data/`。
- 数据库初始化时按目录批量 upsert 默认题目。
- `install.sh` 同步默认测试数据时也支持多个默认题目目录。
- 保持题目 `1000` 作为默认 smoke test 题目。

- [x] **16. 管理页面增加刷新按钮**

管理后台有些页面依赖当前数据库状态，例如用户审核、提交列表、题目管理、语言设置等。管理员完成操作后，如果想确认最新状态，当前主要依赖浏览器刷新，入口不够明确。

后续建议：

- 在管理后台列表类页面增加统一的“刷新”按钮。
- 刷新时保留当前页面的筛选条件、分页参数和排序状态。
- 对用户审核、提交列表这类高频页面优先实现。
- 按钮样式放入统一的后台操作区，避免每个页面单独写一套布局。
- 如果页面后续改为 axios API 化，可以让刷新按钮只重新拉取当前列表数据，而不是整页刷新。

- [x] **17. 排行榜显示用户姓名而不是用户名**

排行榜面向学生和管理员查看学习进度，显示登录用户名不够直观。尤其是用户名可能是学号、拼音或临时账号，无法快速对应到真实学生。

还要有学生的班级

后续建议：

- 排行榜主显示字段改为用户姓名。
- 用户名可以作为次要信息保留，例如放在小字、tooltip 或详情页中，方便管理员排查账号问题。
- 后端排行榜数据需要返回 `displayName` 或等价姓名字段。
- 如果某个用户没有姓名，则回退显示用户名，避免页面空白。
- 排行榜的搜索、排序和班级过滤需要确认是否使用姓名字段。

- [ ] **21. 提交成功后的反馈体验优化**

当前提交代码后会直接跳转到提交详情页。这个流程可用，但用户不一定能明确感知“提交已经成功，正在等待评测”。

后续建议：

- 题目页提交成功后显示 toast 提示，例如“提交成功，正在查看评测结果”。
- 跳转到提交详情页后，在未完成状态下显示更明确的等待提示。
- 防止重复点击提交按钮，提交过程中禁用按钮并显示提交中状态。
- 保持后端提交逻辑不变，只优化前端反馈和页面文案。

- [x] **22. 用户管理增加审核状态和班级筛选**

用户管理当前支持按用户名或姓名搜索，但管理员审核学生时，常见需求还包括只看待审核用户、只看某个班级用户。

后续建议：

- `/admin/users` 增加审核状态筛选：全部、待审核、已通过、已拒绝。
- 增加班级筛选，选项来自数据库班级字典。
- 批量通过、批量拒绝、删除和重置密码后保留当前筛选条件和分页。
- API 管理用户列表同步支持这些筛选条件。
- 增加测试覆盖筛选条件组合和分页保留。

- [ ] **23. 用户管理批量选择当前页**

用户审核页面已经支持批量通过和批量拒绝，但每次仍需要逐个勾选用户。待审核用户较多时，管理员操作效率不够高。

后续建议：

- 在用户管理表格表头增加“选择当前页”复选框。
- 只选择当前页可操作的普通用户，不选择管理员账号。
- 切换搜索、分页或筛选条件后，选择状态不跨页保留。
- 未选择用户时继续使用当前中文提示。
- 增加前端脚本测试或页面行为测试，确保只提交被选中的用户 ID。

- [ ] **24. 列表操作后的筛选和分页上下文统一保留**

部分后台页面已经能在操作后保留筛选和页码，例如用户管理。但题目管理、题目单管理、后台提交刷新等页面还需要统一检查。

后续建议：

- 梳理所有列表类页面的“刷新、发布、隐藏、删除、保存后返回”行为。
- 操作完成后尽量回到原筛选条件和当前页。
- 统一构造列表返回 URL 的 helper，减少每个页面手写 query 拼接。
- 增加测试覆盖关键后台操作后的 redirect location。

## 低优先级

- [ ] **8. 比赛功能占位实现**

当前比赛页面仍然是 placeholder 数据，只用于占位导航和页面结构。

后续如果短期不做比赛功能，建议在页面上明确显示“比赛功能暂未开放”。如果要正式支持比赛，需要单独设计 contest、contest problem、contest submission、contest ranklist 等数据模型。

- [x] **9. 源码阅读文档同步**

部署文档已经同步过，但源码阅读文档中仍然有一些旧路径和旧模块痕迹，例如本机绝对路径、已经删除的旧视图辅助模块等。

后续建议：

- 更新 `docs/source-reading-guide.md` 中的阅读顺序。
- 避免写死 `/home/rainboy/mycode/roj_codex/...` 这类本机绝对路径。
- 删除或改写已经不再推荐阅读的旧模块。
- 增加当前路由拆分后的阅读入口，例如 `routes/`、`http/context.ts`、`packages/db/src/index.ts`。

- [ ] **10. 脚本和文档的一致性检查**

`install.sh`、`quick_start_for_test.sh`、GitHub smoke test、Docker Compose 和 README 都会继续变化。手工同步文档容易遗漏。

后续建议：

- 在测试中检查 README 和 `.env.example` 中的默认端口是否一致。
- 检查文档中提到的默认镜像名是否和 `.env.example` 一致。
- 检查文档中提到的脚本文件是否存在且可执行。
- 对重要部署文档增加最小的静态一致性测试，避免再次出现旧端口或旧镜像名。

- [ ] **13. 排行榜增加班级过滤**

排行榜目前按全站用户汇总，无法只查看某个班级的排名。班级字典化之后，可以把班级作为稳定过滤条件。

后续建议：

- 排行榜页面增加班级筛选控件。
- 后端排行榜查询支持按 `className` 过滤。
- 班级选项来自数据库班级字典，而不是从用户表临时聚合。
- 过滤后仍按现有规则排序：通过题数、错误次数、最后通过时间、用户名。
- URL 使用 query 参数保存筛选状态，方便分享和刷新。

- [ ] **25. 危险操作确认弹窗样式统一**

当前后台危险操作主要依赖浏览器原生 `window.confirm`。功能上可用，但样式和交互不够统一，也不容易承载更详细的中文提示。

后续建议：

- 为删除用户、删除题目单、隐藏题目单等危险操作设计统一确认交互。
- 继续保持轻量实现，不引入大型前端框架。
- 确认文案需要明确说明影响范围和不可恢复风险。
- 失败时继续使用 Notyf 显示错误提示。
